# 要件定義書 - Glean システム全体

## はじめに

Gleanは、エンジニアやAI・IT技術に関心を持つユーザー向けの技術情報キュレーションアプリです。技術ブログやXの投稿などを収集し、OGP画像と概要説明をタイル形式で一覧表示します。タイルを選択すると元の記事や投稿を新しいタブで開くことができます。Gemini APIによる自動要約機能と、タグ・キーワードによる検索機能を提供します。

## 要件

### 要件1: ユーザー認証

**ユーザーストーリー:** ユーザーとして、Google認証またはメールアドレスで登録・ログインしたい。自分だけの記事コレクションを管理できるようにするため。

#### 受け入れ基準

1. WHEN ユーザーがGoogle認証ボタンをクリック THEN システムはFirebase Authentication経由でGoogleログインフローを開始する
2. WHEN ユーザーがメールアドレスとパスワードを入力して登録ボタンを押す THEN システムはFirebase Authenticationでアカウントを作成する
3. WHEN 認証が成功 THEN システムはユーザーをメインページにリダイレクトする
4. WHEN 認証が失敗 THEN システムはエラーメッセージを表示する
5. WHEN ユーザーがログアウトボタンをクリック THEN システムはセッションを終了しログインページに遷移する

### 要件2: 記事・投稿の登録

**ユーザーストーリー:** ユーザーとして、技術ブログやXの投稿のURLを登録したい。後で簡単に見返せるようにするため。

#### 受け入れ基準

1. WHEN ユーザーがURLを入力して登録ボタンを押す THEN システムはURLからOGP情報（タイトル、画像、説明）を取得する
2. WHEN OGP情報の取得が成功 THEN システムはFirestoreにデータを保存する
3. WHEN OGP情報の取得が失敗 THEN システムはURLとタイトルのみで登録し、ユーザーに通知する
4. WHEN 記事が登録される THEN システムはGemini APIを使用して投稿の概要を自動生成する
5. WHEN ユーザーがタグを入力 THEN システムは記事にタグを紐づけて保存する

### 要件3: 記事一覧のタイル表示

**ユーザーストーリー:** ユーザーとして、登録した記事をOGP画像付きのタイル形式で一覧表示したい。視覚的に記事を把握できるようにするため。

#### 受け入れ基準

1. WHEN ユーザーがメインページにアクセス THEN システムは登録済みの記事をOGP画像・タイトル・概要付きのタイル形式で表示する
2. WHEN タイルをクリック THEN システムは元の記事/投稿を新しいタブで開く
3. WHEN 記事が多数ある場合 THEN システムはページネーションまたは無限スクロールで表示する
4. IF OGP画像が存在しない THEN システムはデフォルトのプレースホルダー画像を表示する

### 要件4: 検索機能

**ユーザーストーリー:** ユーザーとして、タイトル・タグ・概要からキーワード検索したい。目的の記事を素早く見つけるため。

#### 受け入れ基準

1. WHEN ユーザーが検索バーにキーワードを入力 THEN システムはタイトル、タグ、Gemini生成概要から一致する記事を表示する
2. WHEN タグをクリック THEN システムはそのタグが付いた記事のみをフィルタリングして表示する
3. WHEN 検索結果が0件 THEN システムは「該当する記事がありません」と表示する
4. WHEN 検索バーをクリア THEN システムは全記事の一覧表示に戻る

### 要件5: タグ管理

**ユーザーストーリー:** ユーザーとして、記事に関連する技術タグを付けて整理したい。カテゴリ別に記事を管理できるようにするため。

#### 受け入れ基準

1. WHEN ユーザーが記事を登録・編集する際にタグを入力 THEN システムはタグを記事に紐づけて保存する
2. WHEN タグ一覧を表示 THEN システムは既存のタグをリスト表示する
3. WHEN ユーザーが既存のタグを選択 THEN システムはそのタグを記事に追加する

### 要件6: Gemini APIによる概要生成

**ユーザーストーリー:** ユーザーとして、登録した記事の内容を自動で要約してほしい。記事の内容を素早く把握できるようにするため。

#### 受け入れ基準

1. WHEN 記事が登録される THEN システムはGemini APIに記事内容を送信し概要を生成する
2. WHEN 概要生成が成功 THEN システムは生成された概要を記事データに保存する
3. WHEN 概要生成が失敗 THEN システムはエラーをログに記録し、概要なしで記事を保存する
4. IF ユーザーが概要を編集 THEN システムは編集内容を保存する

## 非機能要件

### コードアーキテクチャとモジュール性
- **単一責任の原則**: 各ファイルは単一の明確な目的を持つ
- **モジュラー設計**: コンポーネント、ユーティリティ、サービスは分離・再利用可能にする
- **依存関係管理**: モジュール間の相互依存を最小限にする
- **明確なインターフェース**: コンポーネントとレイヤー間のクリーンな契約を定義する

### パフォーマンス
- 初回ページロードは3秒以内
- タイル一覧の表示は1秒以内
- 検索結果の表示は500ms以内
- OGP情報の取得は非同期で行い、UIをブロックしない

### セキュリティ
- Firebase Authenticationによる認証・認可
- Firestoreセキュリティルールでユーザーデータを保護
- Gemini APIキーはサーバーサイドで管理し、クライアントに露出しない
- XSS対策としてユーザー入力をサニタイズする

### 信頼性
- Firebase Firestoreのオフラインサポートを活用
- API呼び出し失敗時のリトライ機構
- エラー発生時のユーザーへの適切なフィードバック

### ユーザビリティ
- レスポンシブデザイン（モバイル・タブレット・デスクトップ対応）
- 直感的なUI/UXデザイン
- ローディング状態の適切な表示
- アクセシビリティ（WCAG 2.1 AA準拠）
